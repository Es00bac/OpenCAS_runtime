Architecture

Explicit subsystems instead of one opaque agent blob.

OpenCAS is organized around runtime concerns: bootstrap, memory, embeddings, autonomy, telemetry, control plane, and channel integration.

Primary subsystems

bootstrap/

Configuration resolution, store wiring, provider material, and runtime assembly.

runtime/

Conversational loop, scheduler, creative cycles, Telegram lifecycle, and background orchestration.

memory/

Episodic storage, distilled memories, graph edges, and artifact-backed memory ingestion.

context/

Session context, retrieval fusion, resonance signals, and memory-backed prompt assembly.

somatic/

Arousal, fatigue, tension, valence, focus, energy, certainty, and modulation logic.

autonomy/

Self-approval, creative ladder, work store, commitments, and executive state.

daydream/

Reflection storage, conflict storage, keeper flow, and promotion lineage.

api/

FastAPI server plus config, chat, memory, operations, usage, identity, executive, and Telegram routes.

dashboard/

Single-page operator UI for inspection, control, and configuration.

Runtime loops

Loop Role
ConversationHandles user input, retrieval, tool use, and response persistence.
Creative cyclePromotes ideas and work objects through the creative ladder.
DaydreamRuns under idle conditions to produce reflections, keepers, and promoted work.
ConsolidationReprocesses memory and long-horizon continuity state on interval.
OperationsTracks PTY, browser, process, qualification, and receipt state.

Persistence layout

The state directory is local and configurable. Under the current CLI, it defaults to ./.opencas.

  • memory.db for episodes, memories, and graph relationships
  • context.db for session message history
  • tasks.db, work.db, plans.db for execution and planning state
  • daydream.db for reflections and conflicts
  • telemetry/ for event traces
  • provider_material/ for app-local OpenLLMAuth config and env material

Provider integration truth

State, dashboards, and operators are local. Model execution is provider-routed by default. Do not describe this architecture as fully local unless you have explicitly replaced the provider-backed model lanes with local equivalents.
