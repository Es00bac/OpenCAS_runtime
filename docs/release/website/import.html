Migration Guide

Bringing your LegacyAgent memory into OpenCAS.

A one-way, checkpointed transfer of your episodes, identity, creative history, and relational state — from LegacyPrototype v4 into a persistent autonomous agent.

What this import is

LegacyPrototype v4 and OpenCAS are two different agents. LegacyAgent was the predecessor — a TypeScript system with its own SQLite and JSONL state. OpenCAS is its Python successor with a richer autonomy stack, relational engine, scheduling, and operator dashboard.

The import is a one-way migration. It reads LegacyAgent's state directory, maps every data category to OpenCAS equivalents, and writes them into the live agent's stores. After the import, OpenCAS boots with your full history, your identity profile, your creative workspace, your daydream sparks, and your relational trust already in place. LegacyAgent's state directory is never modified.

One-way and checkpointed

The import writes into OpenCAS and does not touch LegacyAgent. If it fails partway through, re-running it resumes from the last completed phase — no data is duplicated or lost.

How the data flows

Your LegacyAgent state is first extracted into a clean staging package. You review and prune it. Then the importer reads that package and writes into OpenCAS's live stores.

Your role as operator

You have six steps. The agent handles the rest.

1

Run the extract script

Creates a clean, 65 MB staging package from LegacyAgent's state directory. Runs a preflight scan, copies only the essential categories, redacts any secret-bearing files, and writes REVIEW.md and manifest.json. Takes about 30 seconds. Does not modify LegacyAgent.

python scripts/extract_legacy_agent_package.py --out (external_volumes)/legacy_agent-package
2

Read REVIEW.md

The extract script generates a pruning guide inside the package. It tells you the count of each category, flags duplicate episode IDs (handled automatically), and gives you the exact jq command to thin the 177k memory edges if you want a lighter import.

3

Prune what you don't need

Delete work-product JSON files that are obsolete. Remove workspace sub-directories you don't want migrated. Edit identity/profile.json if the narrative or goals are stale. Filter edges by confidence. None of this is required — skip it if you want everything.

# Optional: keep only edges with confidence > 0.3
jq -c 'select(.confidence > 0.3)' memory/edges.jsonl \
  > memory/edges.filtered.jsonl
mv memory/edges.filtered.jsonl memory/edges.jsonl
4

Start OpenCAS with the server

The importer requires a fully-booted runtime with an active embedding provider. Start OpenCAS normally. The import is triggered through the Python API — not through the dashboard UI.

source .venv/bin/activate
python -m opencas --with-server
5

Call import_legacy_agent()

Point the importer at your staging package. It runs 21 phases, checkpointing after each one. If it fails partway through, re-running resumes from the last completed phase. The embedding provider must be available — this is where Kimi (or another configured model) is used to embed imported episodes.

# In a Python session with the venv active:
from opencas.runtime.agent_loop import AgentRuntime
from pathlib import Path
import asyncio

# runtime = your running AgentRuntime instance
report = asyncio.run(runtime.import_legacy_agent(
    legacy_agent_state_dir=Path("(external_volumes)/legacy_agent-package"),
))
print(report.model_dump_json(indent=2))
6

Verify

Check the import report for counts. Open the Memory tab in the dashboard and confirm episodes are present. Check the Identity tab for your narrative and values. The cutover manifest at .opencas/cutover_manifest.json lists every file that was processed with its sha256 hash.

What each category becomes

💬

Episodes

Each LegacyAgent episode becomes an OpenCAS Episode in memory.db, assigned a new UUID while preserving the original LegacyAgent ID for deduplication. Source, content, salience, emotion tags, and identityCore flag are all mapped.

→ memory.db / episodes table
🔗

Memory edges

LegacyAgent's semantic, emotional, recency, and salience weights are averaged into OpenCAS EpisodeEdge confidence scores. Orphan edges (pointing to missing episodes) are skipped with a count in the report.

→ memory.db / episode_edges table
🪪

Identity

Narrative, values, traits, and ongoing goals map to the OpenCAS SelfModel. Partner trust and musubi scores seed the relational engine. Recent themes and memory anchors are preserved on the self-model for context.

→ identity/ + relational.db
🌙

Daydreams

Sparks, initiatives, outcomes, and notifications become DaydreamReflection objects in the daydream store. Keeper daydreams with a track record of value delivered are flagged accordingly.

→ daydream.db
📂

Workspaces

Creative workspace directories (chronicle chapters, scripts, research files) are copied into OpenCAS workspace storage. A workspace manifest is created for each top-level project directory.

→ workspace storage + work.db
📝

Work products

Each LegacyAgent work-product draft JSON becomes a WorkObject entry in the creative ladder's work store, with objective, summary, and status carried through.

→ work.db
🌡️

Somatic history

The most recent somatic snapshot (arousal, fatigue, valence, certainty) seeds the OpenCAS somatic store so the agent's starting emotional state reflects recent LegacyAgent history rather than cold defaults.

→ somatic.db
🎯

Executive state

Active goals and commitments become OpenCAS Goal and Commitment records. Executive events are archived and indexed rather than imported verbatim — too large to carry as live state.

→ commitments.db + executive archive

Why embeddings matter for this import

The embeddings phase is the most expensive one. Every imported episode needs a vector embedding so OpenCAS can retrieve it semantically — so you can ask the agent about something and it can find relevant memories even if the exact words don't match.

Those embeddings are computed during import by calling the configured embedding provider (google/gemini-embedding-2-preview routed through open_llm_auth).

Why the import is waiting

Kimi usage resets on an 11-hour cycle. Once it resets, the embedding provider is available and the import can run with full semantic vector quality. The staging package at (external_volumes)/legacy_agent-package is already prepared and ready — no further extract steps are needed.

Safety and rollback

🔒

LegacyAgent is never touched

The extract script and importer are both read-only with respect to LegacyAgent's state directory. Nothing in (legacy_path)/state is modified or deleted.

♻️

Checkpointed resumption

A checkpoint file tracks completed phases. Re-running the importer after a failure picks up exactly where it left off — no phase is repeated, no data duplicated.

🗑️

Simple rollback

Because the import writes into a fresh OpenCAS instance, rolling back means wiping .opencas/ and re-running. The staging package is reusable.

Audit trail

The cutover manifest at .opencas/cutover_manifest.json lists every source file that was considered, its disposition (imported, retired, skipped), size, and sha256. The import report printed at the end gives per-phase counts for everything that was written.
