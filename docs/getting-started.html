1. Prerequisites

  • Python 3.11+ (3.12 or 3.13 recommended; 3.14 disables HNSW)
  • open_llm_auth installed as an editable dependency (already wired in requirements.txt)
  • At least one LLM provider API key (Anthropic, Google, Kimi, OpenAI, etc.)
  • Optional: Qdrant server for production-grade vector search
1 Activate the virtual environment and install dependencies
bash 
cd /path/to/OpenCAS
source .venv/bin/activate
pip install -r requirements.txt
2 Verify the install
bash 
python -m opencas --help

You should see the full argument list. If you get an import error, ensure open_llm_auth is installed:

bash 
pip install -e /path/to/open_llm_auth/

2. Set Up LLM Credentials

OpenCAS uses open_llm_auth as its LLM gateway. It needs a provider config with at least one working auth profile. There are three approaches:

Option A: Use your existing OpenLLMAuth config (simplest)

If you already have ~/.open_llm_auth/config.json with profiles configured, OpenCAS will find it automatically. No extra flags needed.

Option B: Copy specific credentials into app-local state

This copies selected profiles from your global config into the agent's own state directory, isolating it from your main config:

bash 
python -m opencas \
  --with-server \
  --credential-source-config-path ~/.open_llm_auth/config.json \
  --credential-profile-id "anthropic:claude-code-pro" \
  --credential-profile-id "kimi-coding:default" \
  --credential-env-key "GOOGLE_API_KEY"

Option C: Point to a custom config directly

bash 
python -m opencas \
  --with-server \
  --provider-config-path /path/to/my/config.json \
  --provider-env-path /path/to/my/.env
Embedding model

By default, OpenCAS uses google/gemini-embedding-2-preview for embeddings, which requires a Google API key. If you don't have one, override to local-only embeddings:

bash 
python -m opencas --with-server --embedding-model-id local-fallback

3. First Boot: Start the Agent

The minimum command to start OpenCAS with the dashboard and API server:

bash 
cd /path/to/OpenCAS
source .venv/bin/activate

python -m opencas --with-server

This will:

  1. Create the state directory at ./.opencas/ (configurable via --state-dir)
  2. Run the full bootstrap pipeline (identity, memory, embeddings, somatic, relational, ToM, plugins, etc.)
  3. Start the FastAPI server on http://127.0.0.1:8080
  4. Start the autonomous background scheduler (creative cycles, daydreaming, BAA heartbeat, consolidation)
First boot is special

On first boot (boot_count == 0), OpenCAS seeds default identity values, emits a moral-warning trace, and initializes the relational field at baseline. Subsequent boots restore existing state.

Full production-style launch example

bash 
python -m opencas \
  --with-server \
  --state-dir ~/.opencas/agents/main \
  --workspace-root /path/to/OpenCAS \
  --workspace-extra-root /path/to/open_llm_auth \
  --session-id "session-$(date +%Y%m%d-%H%M%S)" \
  --default-llm-model "anthropic/claude-sonnet-4-6" \
  --embedding-model-id "google/gemini-embedding-2-preview" \
  --host 127.0.0.1 \
  --port 8080 \
  --cycle-interval 300 \
  --consolidation-interval 86400

Headless mode (no server, background loops only)

Omit --with-server to run only the autonomous scheduler. The agent will cycle, daydream, and run BAA tasks but won't expose a dashboard or API:

bash 
python -m opencas --state-dir ~/.opencas/agents/main

4. Understanding the State Directory

All agent state lives under the --state-dir path. On a fresh boot, the pipeline creates:

.opencas/ ├── memory.db ← episode storage (conversations, events) ├── embeddings.db ← vector cache (computed embeddings) ├── context.db ← session conversation history ├── tasks.db ← BAA task queue and transition records ├── receipts.db ← execution receipt audit trail ├── work.db ← creative ladder work objects ├── relational.db ← musubi state and history ├── tom.db ← theory of mind beliefs/intentions ├── plans.db ← structured plans ├── commitments.db ← durable goal commitments ├── portfolio.db ← portfolio clusters ├── daydream.db ← daydream reflections ├── conflict.db ← conflict registry ├── governance.db ← approval ledger ├── plugins.db ← plugin/skill registry ├── harness.db ← research notebooks, objective loops ├── curation.db ← consolidation curation ├── somatic.db ← somatic snapshot history ├── somatic.json ← live somatic state ├── executive.json ← executive state snapshot (goals, intention) ├── identity/ ← self-model, user-model, continuity ├── self_knowledge.jsonl ← structured self-beliefs ├── telemetry/ ← JSONL event traces and token usage └── provider_material/ ← copied auth credentials (if using --credential-*)
Multiple agents

You can run multiple agents with separate state dirs. Each one is fully isolated:

bash 
# Agent A on port 8080
python -m opencas --with-server --state-dir ~/.opencas/agents/alpha --port 8080

# Agent B on port 8081 (separate terminal)
python -m opencas --with-server --state-dir ~/.opencas/agents/beta --port 8081

5. Using the Dashboard

Once the server is running, open your browser:

url 
http://127.0.0.1:8080/dashboard

The dashboard has six tabs:

TabWhat It Shows
OverviewAgent identity, readiness state, somatic state, executive snapshot, active goals, musubi score, qualification summary
HealthDoctor health checks (memory, embeddings, BAA queue depth, compaction lag), health check history
ChatInteractive chat interface — send messages and see responses directly in the browser
OperationsPTY sessions, browser sessions, background processes, execution receipts, work items, commitments, plans, qualification, validation runs
MemoryEpisode browser, memory graph, search, embedding projection, memory stats
SystemProvider config, runtime config, model info

Chat in the Dashboard

Go to the Chat tab. Type a message and press Enter (or click Send). The agent will process your input through:

  1. The conversational refusal gate (safety check)
  2. Somatic appraisal of your input
  3. Context building (system prompt + retrieved memories + somatic style notes + relational/ToM directives)
  4. LLM tool-use loop (conversation + tool calls if needed)
  5. Memory recording, ToM belief tracking, relational state update

6. Terminal Chat via API

You can interact with the agent from the terminal using curl:

Send a message

bash 
curl -s http://127.0.0.1:8080/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "Hello, what can you do?"}' | python3 -m json.tool

Send a message to a named session

bash 
curl -s http://127.0.0.1:8080/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "List the files in the project root", "session_id": "work-session-1"}' \
  | python3 -m json.tool

Build a simple terminal chat loop

bash 
#!/usr/bin/env bash
# opencas-chat.sh — simple terminal chat client
SESSION="terminal-$(date +%s)"
echo "OpenCAS Chat (session: $SESSION). Type 'quit' to exit."
echo "---"
while true; do
  printf "\033[1;36mYou:\033[0m "
  read -r input
  [ "$input" = "quit" ] && break
  response=$(curl -s http://127.0.0.1:8080/chat \
    -H "Content-Type: application/json" \
    -d "{\"message\": $(echo "$input" | python3 -c 'import sys,json; print(json.dumps(sys.stdin.read().strip()))'), \"session_id\": \"$SESSION\"}" \
    | python3 -c 'import sys,json; print(json.load(sys.stdin).get("response","[no response]"))')
  printf "\033[1;32mCAS:\033[0m %s\n\n" "$response"
done

7. WebSocket Live Chat

For real-time streaming and event subscriptions, connect via WebSocket:

bash 
# Using websocat (install: pacman -S websocat)
websocat ws://127.0.0.1:8080/ws

Then send JSON messages:

json 
{"type": "chat", "payload": "What are your current goals?", "session_id": "ws-session"}

The server responds with:

json 
{"type": "chat_response", "session_id": "ws-session", "payload": "..."}

The WebSocket also receives live telemetry events pushed by the EventBus, so you can watch bootstrap stages, tool calls, somatic shifts, and creative promotions in real time.

8. Give the Agent Work

The agent can perform work through several channels:

Conversational work (via chat)

Simply tell the agent what to do. It has access to filesystem, shell, PTY, browser, planning, and workflow tools:

json 
{"message": "Read the file opencas/somatic/models.py and summarize the SomaticState fields"}
json 
{"message": "Create a plan for refactoring the memory retrieval pipeline"}
json 
{"message": "Run pytest tests/test_memory.py and report the results"}

Setting goals (drives autonomous behavior)

The agent extracts goal directives from natural language. Phrases like "I want to..." or "Our goal is..." are auto-detected:

json 
{"message": "Our goal is to improve test coverage for the somatic module"}

You can also manage goals via the API:

bash 
# View runtime state including active goals
curl -s http://127.0.0.1:8080/api/monitor/runtime | python3 -m json.tool

Workflow tools (higher-level operations)

Ask the agent to use its workflow tools for structured work:

json 
{"message": "Create a writing task: write a technical summary of how memory retrieval works in OpenCAS"}
json 
{"message": "Triage this repo and tell me what's been happening recently"}

10. Operations Control Plane

The Operations tab (and /api/operations/*) gives you direct control over running sessions, work items, plans, and commitments.

Sessions (PTY, Browser, Background Processes)

bash 
# List all active sessions
curl -s http://127.0.0.1:8080/api/operations/sessions | python3 -m json.tool

# Inspect a specific PTY session
curl -s http://127.0.0.1:8080/api/operations/sessions/pty/{session_id} | python3 -m json.tool

# Send input to a PTY session
curl -s http://127.0.0.1:8080/api/operations/sessions/pty/{session_id}/input \
  -H "Content-Type: application/json" \
  -d '{"input": ":wq\n"}' | python3 -m json.tool

# Kill a PTY session
curl -s -X DELETE http://127.0.0.1:8080/api/operations/sessions/pty/{session_id}

# Bulk cleanup
curl -s -X DELETE "http://127.0.0.1:8080/api/operations/sessions/pty?scope_key=default"

11. Supervised Readiness Checklist

For your first session, walk through this checklist to confirm the agent is operating correctly:

#CheckHow to Verify
1 Server is healthy curl http://127.0.0.1:8080/health returns {"status":"ok"}
2 Readiness is READY curl http://127.0.0.1:8080/readiness shows "state": "ready"
3 Doctor health passes Dashboard Health tab shows all green, or GET /api/monitor/health
4 LLM provider works Send a simple chat message. If you get [Error...], check your credentials.
5 Embeddings compute GET /api/monitor/embeddings — should show a model_id and cache hit count
6 Identity seeded Dashboard Overview shows agent name and boot count ≥ 1
7 Somatic state live GET /api/monitor/runtime — somatic section shows non-default values after a few messages
8 Memory records turns After chatting, GET /api/memory/stats shows episode count > 0
9 Tool use works Ask it to "List files in the current directory" — verify it returns real file names
10 No stale processes GET /api/operations/sessions — process/pty/browser counts should be 0 at startup
If all 10 pass

The agent is in a good supervised-readiness state. You can now use it for real work, knowing the substrate is healthy.

If LLM calls fail

The most common issue. Check: (1) your ~/.open_llm_auth/config.json has valid, non-expired credentials, (2) the model you're targeting is in the provider catalog, (3) the --default-llm-model matches an available profile's provider.

12. Shutdown and Restart

Graceful shutdown

Press Ctrl+C in the terminal running OpenCAS. This triggers:

  1. Signal handler sets the shutdown event
  2. Readiness moves to SHUTDOWN
  3. FastAPI server stops accepting new requests
  4. Scheduler stops all background loops
  5. BAA drains and stops
  6. All stores are closed cleanly

Restart with existing state

Just start again with the same --state-dir. All state is durable. The boot count increments, identity is restored, executive goals are rehydrated, ToM beliefs and intentions are reloaded from SQLite, and the relational field picks up where it left off.

bash 
# Restart with existing state
python -m opencas --with-server --state-dir ~/.opencas/agents/main

Fresh start (clean boot)

To wipe identity and start over without deleting databases:

bash 
# Delete the state directory entirely for a true fresh start
rm -rf ./.opencas

# Or just re-seed identity by removing the identity dir
rm -rf ./.opencas/identity

Kill stale background processes

If you had PTY/process sessions running when you killed the agent hard:

bash 
python scripts/sweep_operator_processes.py --kill

13. Troubleshooting

"Missing critical dependency: open_llm_auth"

bash 
pip install -e /path/to/open_llm_auth/

"OpenCAS requires Python >= 3.11"

Your venv is using an older Python. Recreate it:

bash 
python3.12 -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt

Chat returns "[Error: ...]"

  1. Check GET /api/config to see what model is configured
  2. Check GET /api/config/providers to see if auth profiles loaded
  3. Look at the terminal output for stack traces
  4. Verify your API key or OAuth token hasn't expired

Embeddings falling back to local-fallback

You'll see model_id: local-fallback in the embedding status. This means:

  • No Google API key was found, or
  • You explicitly set --embedding-model-id local-fallback

Local fallback uses a 256-dim deterministic hash. It works, but semantic search quality will be low. For production use, configure a real embedding model.

Dashboard shows but is empty

The dashboard uses Alpine.js and fetches data from the API. If the Overview is blank, check the browser console (F12) for fetch errors. The most common cause is the API returning errors because stores failed to connect.

Telemetry / debugging

All events are logged as JSONL files in the telemetry directory:

bash 
# Tail the latest telemetry events
tail -f .opencas/telemetry/*.jsonl | python3 -m json.tool --no-ensure-ascii
